本产品一共有三个模块，第一个为计划，用于生成计划和使用。第二个为打卡，用于真实记录用户的训练内容。第三个为历史，用于存储用户的记录。
以下为第一个板块，有关计划的内容
# 计划板块
规则：对于时间，数据库一律记录秒数，但是前段现实的时候要按照hh：mm：ss的格式。
对于距离，数据库一律记录公里，但是在前端显示的时候，对于1km一下的距离显示为m，比如800m，而不是0.8k。
### user
user在本地，记录用户自己本身的数据，比如跑步目标、睡眠（起床、睡觉）目标、体重目标
跑步目标的话，会有一个对应的vdot value

## plan相关设计
### unit 
unit < dailyPlan < weeklyPlan < Plan
unit用于记录最小单元，三种类型。跑步、休息和其他。
跑步类型：需要两个字段：配速和标准。

配速有两种模式：一种为按照vdot的五种配速区间的，另外一种为自定义。当用户有选定plan的时候，优先按照plan的vdot。如果用户没有选择plan，用用户本身user表内存在的vdot。
所以在记录跑步类型的记录的时候，如果用户选择使用vdot模式，用户的配速为E，M，T，I，R，E+10，R-5等这样，用户可以在此基础上进行一些调整。如果使用hardcode模式，就是手动直接输入配速。按照mm:ss格式读入。
标准有两种：时间或者距离。

休息的时候，需要一个字段，即标准。标准有两种：时间或者距离。
其他：需要一个字段，即内容。内容为text文本，用户自行输入

### dailyPlan
dailyplan是对unit的有序排列组合。这个dailyPlan不仅是一个记录unit的列表，在之后用户打卡运动的时候也是创建dailyWorkout的部分。要有个desc字段，用于描述一天训练的大概内容。

### weeklyPlan
主要目的是记录本周的计划描述。要有个desc字段，用于描述一周训练的大概内容。
### Plan
plan为长期的计划，会包括target dis和time，以及计划持续的周数。在有了target dis和time后，我们可以计算出来vdot，靠这个vdot我们可以记录这个plan的五个配速区间。里面应该可以记录week index。要有个desc字段，用于描述这个训练的大概内容。

# 使用计划
用户可以选择使用计划。使用计划后，会选择从下周开始还是本周开始。一旦选择后，就会在user_plan_calendar当中开始记录此用户的计划日历了。这一步是把单纯的计划内容和真实日期联系起来。一旦用户选择使用计划了，便代表后续的处理都是基于user_plan_calendar的了。

# 打卡模块
这里需要一个数据库表，为记录今日打卡内容的，要有一个bool状态。默认false，当有新的记录进入时候，变成true，这个设计在前端有用处

在打卡板块，主要目的是记录用户真实完成了什么内容。
如果用户使用了计划，当我们点击打卡后，可以选择记录是否完成了今日计划。
如果是，可以把这一条加入今天的打卡记录。
如果不是，用户可以在这里创建今天的训练内容。对应到刚才说的部分（dailyplan是对unit的有序排列组合。这个dailyPlan不仅是一个记录unit的列表，在之后用户打卡运动的时候也是创建dailyWorkout的部分。要有个desc字段，用于描述一天训练的大概内容。）
用户可以多次打卡，对应着用户一天可能有多次训练，但是只要有一次打卡，bool就会变成true

在打卡的时候，可以对于训练内容，需要记录的内容为训练的内容，感受（option），图片（option）

# 记录模块

在这里就是按照日历，存储每一天的打卡内容，并且可以进行查看

以上为最核心的功能，涉及了训练、打卡、记录这件事情的全链路。在后期，我们会加入体重记录和睡眠的记录。但是目前我们不需要过度关心，基本的逻辑是一样的。
